私のMacBookでローカルエージェントを動かしていると、会話が長くなるほど応答が目に見えて鈍くなる。「体感で遅い」のは分かっていたが、どの段階がどれだけ遅いのかは知らなかった。そこでOllamaが応答ごとに返すタイムスタンプを直接取り出して測ってみた。

結論から書く。同じ9,700トークンのプロンプトを2回送ったら、1回目は最初のトークンが出るまで約55秒、2回目は65ミリ秒で処理された。同じ入力なのに約396倍の差だ。この一行が、ローカルLLMの遅延のほとんどを説明する。

## Ollamaが応答に隠したストップウォッチ

多くの人はOllamaをチャットUIか`ollama run`でしか触らない。だが`/api/generate`を`stream:false`で呼ぶと、応答JSONに精密なタイミングフィールドが一緒に返ってくる。[Ollama APIドキュメント](https://github.com/ollama/ollama/blob/main/docs/api.md)に明記されたフィールドだ。

- `prompt_eval_count`: 入力プロンプトのトークン数
- `prompt_eval_duration`: プロンプトを処理するのにかかった時間(= prefill)
- `eval_count`: 生成したトークン数
- `eval_duration`: トークンを生成するのにかかった時間(= generation)
- すべてのdurationはナノ秒単位

ここで肝心なのは、LLM推論が性質のまったく違う2段階に割れることだ。<strong>prefill</strong>は入力したプロンプト全体を一気に読んでKVキャッシュを埋める段階で、最初のトークンが出る直前までがここに入る。<strong>generation</strong>はその後、トークンを1個ずつ自己回帰的に吐く段階だ。両者はコスト構造が違い、コンテキストが伸びたときの反応も違う。それを分けて見たかった。

測定スクリプトは標準ライブラリだけで短く書いた。コンテキスト長を変えながら同じ質問を投げ、応答のタイミングフィールドをトークン/秒に換算する。

```python
import json, urllib.request

OLLAMA = "http://localhost:11434/api/generate"
MODEL  = "gemma4:e4b"

def call(prompt):
    body = json.dumps({
        "model": MODEL, "prompt": prompt, "stream": False,
        "options": {"num_predict": 64, "seed": 42, "temperature": 0}
    }).encode()
    req = urllib.request.Request(OLLAMA, data=body,
                                headers={"Content-Type": "application/json"})
    with urllib.request.urlopen(req, timeout=900) as r:
        d = json.load(r)
    pe_n, pe_d = d["prompt_eval_count"], d["prompt_eval_duration"] / 1e9
    ev_n, ev_d = d["eval_count"], d["eval_duration"] / 1e9
    return {
        "ctx": pe_n,
        "prefill_s": pe_d,
        "prefill_tps": pe_n / pe_d,   # prefillの処理速度
        "gen_tps": ev_n / ev_d,       # 生成速度
    }
```

先に一つの罠を処理する必要があった。同じプロンプトを繰り返すと、キャッシュのせいでprefillがほぼ0になる(後述)。「冷えた(cold)」prefillを正直に測るには、各呼び出しのプロンプトが<strong>先頭の1バイト</strong>から違わなければならない。そこでプロンプトの頭に毎回違う乱数IDを付け、本文もシードを変えて埋めた。測定モデルは軽く回すために`gemma4:e4b`を選び、最初の呼び出しの前に一度ウォームアップして、モデルのロード時間(`load_duration`)が測定に混ざらないようにした。

## コンテキストが長いほど最初のトークンが遅れる

まず冷えたprefillから。コンテキストを約200トークンから9,700トークンまで伸ばし、最初の出力トークンまでの時間を測った。

| コンテキスト(トークン) | cold prefill | prefill tok/s | generation tok/s | 1トークン生成(ms) |
|---:|---:|---:|---:|---:|
| 200 | 1.0秒 | 197.8 | 16.81 | 59.5 |
| 644 | 3.2秒 | 198.4 | 16.61 | 60.2 |
| 1,244 | 6.3秒 | 197.6 | 16.38 | 61.1 |
| 2,476 | 12.7秒 | 194.6 | 16.40 | 61.0 |
| 4,852 | 25.7秒 | 188.6 | 15.87 | 63.0 |
| 9,716 | 54.8秒 | 177.3 | 15.43 | 64.8 |

![コンテキスト長に対するprefill時間と生成速度](../../../assets/blog/local-llm-prefill-generation-latency-experiment/results-chart.png)

左のグラフはほぼ直線だ。コンテキストが48倍(200 → 9,716)になると、prefillが1秒から55秒へ約54倍に伸びた。ほぼ比例する。直感的には当たり前だ。トークンが多いほど読むものが多い。

面白いのはprefillの<strong>速度</strong>(tok/s)だ。コンテキストが短いときは198 tok/sだったのに、1万トークン近くでは177 tok/sへ約10%落ちた。つまりトークン1個を処理するコスト自体が、コンテキストが長いほど高くなる。私の理解では、アテンションがシーケンス長に対して二乗で効くためだ。短い文で単語1個を読むより、長い文の末尾で前全体を再参照しながら単語1個を読む方が重い。だからprefillは「トークン数に比例」より少し急に伸びる。

ここで最初の実践的な教訓。ローカルエージェントの応答が遅い主犯は、たいてい生成速度ではなくprefillだ。RAGで文書を5つ詰め込んだり、直前の会話20ターンを丸ごと送り直したりすると、モデルが最初の一文字を書く前に数十秒が飛ぶ。クラウドAPIではこのコストが[データ形式によってトークン数が変わり課金される](/ja/blog/ja/llm-token-cost-data-format-experiment)が、ローカルではそっくり私の時間として請求される。

ストリーミングUIを使うなら、このprefill時間はそのままユーザーが空白画面やローディングスピナーを見つめる時間だと思えばいい。トークンが流れ出せば、その後は毎秒16個でそこそこ速く埋まる。問題は最初の一文字までだ。コンテキストが長いほど、ユーザーは「固まったのか?」と感じる沈黙を長く耐えることになる。ローカルチャットボットの体感応答性を左右するのは、トークンが流れる速さではなく、この沈黙の長さだ。9,700トークンのコンテキストで55秒間なにも出なければ、それは対話的に使える道具ではない。

## 生成速度もこっそり落ちる

右のグラフは変化が小さくて見過ごしやすい。だからこそ取り上げたい。生成速度が16.81 tok/sから15.43 tok/sへ約8%落ちた。トークン1個を吐く時間に直すと59.5msから64.8msに伸びた。

なぜか。生成段階で新しいトークンを1個作るたびに、モデルは前に積んだKVキャッシュ全体をアテンションで再び舐める。コンテキストが長ければ舐める対象が多くなるので、トークンあたりの時間が少しずつ伸びる。これもアテンションが長さに敏感だという同じ理由だ。

ただ正直に言うと、この8%はprefillの前では枝葉だ。1万トークンのコンテキストで64トークンを生成するのに約4秒かかるが、その前段のprefillは55秒だった。体感の遅延の9割以上は最初のトークンの前で起きている。だから私は「生成をもっと速いモデルに替えよう」より「プロンプトを短く、そしてキャッシュされるようにしよう」のほうがはるかに大きなレバーだと見ている。

## 2回目の呼び出しが396倍速かった理由

この実験で一番印象的だったのはキャッシュだ。4,859トークンの同一プロンプトを連続で2回送った。

| 呼び出し | prompt_eval_count | prefill時間 |
|---|---:|---:|
| 1回目 (cold) | 4,859 | 25,751ms |
| 2回目 (warm) | 4,859 | 65ms |

`prompt_eval_count`は2回とも4,859で同じだった。トークン数はそのままなのに、prefill時間だけ約396倍縮んだ。トークンを「読まなかった」のではなく、すでに計算済みのKVキャッシュを再利用して、再計算する必要がなかったのだ。

これが[llama.cppのprefix KVキャッシュ](https://github.com/ggml-org/llama.cpp/discussions/13606)だ。Ollamaはllama.cppの上で動くので、同じ挙動を受け継ぐ。先頭(prefix)が同じ2つのリクエストは、その区間のKVキャッシュがビット単位で同じなので、2回目は共通の接頭を飛ばし、違いが出る地点からだけ処理する。同一プロンプトなら違いが出る地点がないので、prefillが実質消える。

一つ戸惑った点も書いておく。キャッシュが当たっても`prompt_eval_count`は依然として4,859という満杯の値を返す。最初はこの数字だけ見て「キャッシュが効いていないのか」と思ったが、見るべきはトークン数ではなく`prompt_eval_duration`だった。キャッシュが効いているか確認するには、countではなくprefill時間を見る。同じプロンプトを2回送り、2回目のprefillが数ミリ秒に落ちれば当たりだ。ここを取り違えると、キャッシュが効く環境でも「キャッシュされない」と誤診しやすい。

さっき冷えたprefillを測るためにプロンプトの頭に乱数IDを付けたと書いた。その理由が今はっきりする。キャッシュは<strong>先頭から共通の区間</strong>だけ再利用する。最初の1バイトが変われば、その後は全部計算し直しになる。つまりプロンプトの先頭に毎リクエスト変わる値を置くと、キャッシュが丸ごと壊れる。

## ではエージェントをどう組むべきか

この測定をしてから、私はローカルエージェントのプロンプト構成を変えた。整理するとこうだ。

<strong>1. 変わらないものを前に、変わるものを後ろに。</strong> システムプロンプト、ツール定義、固定の指示のように毎ターン同じ内容はプロンプトの先頭に置く。ユーザーの新しい質問やさっき検索した結果のように毎回変わる内容は後ろへ回す。これだけで2ターン目から先頭のprefillがキャッシュに乗り、ほぼ無料になる。

<strong>2. プロンプトの頭にタイムスタンプ・ランダムIDを入れない。</strong> 「現在時刻: 2026-06-25 15:23:06」のような行をシステムプロンプトの最上段に埋め込むと、毎リクエストで最初の行が変わってキャッシュが毎回壊れる。どうしても入れるならプロンプトの末尾へ送る。このディテール一つでprefillを数十秒単位で節約できる。

<strong>3. コンテキストウィンドウが「入る」のと「使える」のは違う。</strong> モデルが32kに対応していても、私のノートPCでは1万トークンで最初のトークンまで55秒だった。対話的に使うなら、対応上限ではなく測定したprefill時間を基準にコンテキスト予算を組むべきだ。私はローカルで対話的な応答を狙うときはコンテキストを数千トークン以内にまとめる。

<strong>4. 長いコンテキストがどうしても必要なら、prefillを1回だけ払って再利用する。</strong> 同じ文書に複数の質問を投げるRAGなら、文書をプロンプト前方の固定位置に置き、質問だけ後ろで変える。最初の質問はprefillを全額払うが、続く質問はキャッシュのおかげでprefillがほぼ消える。[ローカルモデルをMCPサーバーに繋いでエージェントを組むとき](/ja/blog/ja/local-llm-private-mcp-server-gemma4-fastmcp)も、この順序を守ればツール呼び出しの往復ごとにシステムプロンプトを再prefillせずに済む。

## 実際に変えた私のプロンプト

抽象的なルールだけではピンと来ないので、私のローカルなツール呼び出しエージェントのプロンプトをどう並べ替えたかを書いておく。変える前はこの順序だった。

```text
現在時刻: 2026-06-25 15:23:06        ← 毎リクエスト変わる(キャッシュ破壊点)
セッションID: 9f3a-...               ← 毎リクエスト変わる
[システムプロンプト 800トークン]
[ツール定義 1,200トークン]
[直前の会話履歴]
[ユーザーの新しい質問]
```

問題は最初の2行だ。時刻とセッションIDが先頭にあるので、リクエストごとにプロンプトの最初の文字が変わる。その後ろのシステムプロンプト800トークンとツール定義1,200トークンは毎ターン一字も変わらないのに、前でキャッシュが壊れるせいで、毎回最初からprefillをやり直していた。上の表によれば2,000トークンのprefillは約10秒。毎ターン10秒ずつ、純粋に配置ミスで捨てていたわけだ。

変えた後はこうした。

```text
[システムプロンプト 800トークン]      ← 固定、先頭
[ツール定義 1,200トークン]            ← 固定
[直前の会話履歴]                      ← ほぼ固定(末尾に追加されるだけ)
現在時刻: 2026-06-25 15:23:06         ← 変わる値は末尾へ
セッションID: 9f3a-...
[ユーザーの新しい質問]                 ← 毎回変わる部分は最後
```

固定ブロックを前にまとめると、2ターン目からシステムプロンプトとツール定義の2,000トークンが丸ごとキャッシュに乗った。直前の会話履歴も末尾にだけ新しいメッセージが付く構造なので、共通の接頭が長く保たれる。結果、毎ターン再prefillが必要なのは新しく足された数百トークンだけになった。同じモデル、同じハードウェアなのに、ターンあたりの体感遅延が目に見えて減った。コードは1行も速くなっていない。ただ文字列を積む順序を変えただけだ。

## この測定の限界

正直に境界を引いておく。これはMacBook1台、モデル1つ(`gemma4:e4b`)、Ollamaという特定のランタイムで出た数字だ。絶対値(55秒、16 tok/s)はGPU・メモリ・量子化・ランタイムによって丸ごと変わる。より大きなモデルや[出力を型で受け取る構造化出力](/ja/blog/ja/ollama-structured-outputs-pydantic-local-llm-guide-2026)を使えば、また別の変数が入る。

私が信頼するのは絶対値ではなく<strong>形</strong>だ。prefillはコンテキストにほぼ比例して伸び、生成は少し遅くなり、同一prefixはキャッシュでほぼ無料になる。この3つの傾向はどの環境でも向きが同じだと見ている。同時リクエストが入るときキャッシュがどう競合するか、量子化レベルがprefill速度に与える影響はまだ測っていない。次の実験のネタとして残しておく。

ローカルLLMを「遅い/速い」で一括りにせず、最初のトークンまでのprefillとトークンあたりのgenerationに割って見ると、どこを直せばいいかがはっきりする。そしてその直しどころの大半は、モデルを替えることではなく、プロンプトをキャッシュされるように並べることだった。

今回の実験で得た一番実用的な一行はこれだ。ローカルでエージェントを組むとき、最初に点検すべきはGPUでもモデルサイズでもなく、「自分のプロンプトの先頭が毎ターン同じまま保たれているか」だ。先頭を固定するだけで、同じハードウェアでも2ターン目からprefillがほぼ消える。お金も追加のGPUも一切かけずに得られるタダの加速だ。測るまで、私はこれをただの「体感」として片付けて見過ごしていた。
