エージェントに商品カタログ50行をコンテキストとして渡す必要があった。いつもの癖で `json.dumps(records, indent=2)` ときれいに整形して入れたら、トークンカウンタが4,000を超えていた。データ自体は小さい。インデントとクォートがトークンの半分近くを食っているのではと疑った。そこで同じデータを9形式に直列化し、実際のトークナイザで数えてみた。

先に結論を言う。**平坦なデータならTSVはpretty JSONより62%安い。** ところがデータがネストした瞬間、この結論は丸ごと逆転する。その境界が実際どこにあるのか、自分で測りながら整理した。

## 計測環境：推測でなくtiktokenで数える

トークンコストは「だいたい文字数 × 0.75」と語られがちだが、その目安では形式ごとの差はまったく拾えない。同じ目安が言語を変えるとどう崩れるかは、[同じ記事でも韓国語はトークンが1.4倍かかる非英語トークン税の実測](/ja/blog/ja/multilingual-llm-token-tax-experiment)で別途測った。だからOpenAIが公開している [tiktoken](https://github.com/openai/tiktoken) をそのまま使った。GPT-4o・o系・GPT-5系が使う `o200k_base` と、旧GPT-4系の `cl100k_base` の2つを並べて回した。

テストデータは現実的な「ツール結果」を模した。商品レコード50件、それぞれ `id`・`sku`・`name`・`category`・`price`・`stock`・`warehouse`・`status`・`rating` の9フィールドを持つ平坦な配列だ。

```python
import json, io, csv, yaml, tomli_w, tiktoken

records = json.load(open("records.json"))  # 50件の平坦レコード
enc = tiktoken.get_encoding("o200k_base")  # GPT-4o / GPT-5系

def tok(s): return len(enc.encode(s))

print("pretty :", tok(json.dumps(records, indent=2)))
print("compact:", tok(json.dumps(records, separators=(",",":"))))
print("csv    :", tok(to_csv(records)))
```

サンドボックスはrepo外の使い捨て `mktemp -d` ディレクトリで回し、結果ログとチャートだけ残して環境は消した。こうした1回限りの検証環境を分離する習慣は、[エージェントを8体動かしてコストを検証した経験](/ja/blog/ja/ai-agent-cost-reality)で固まったものだ。トークンを削るのは結局、その費用台帳の一行である。

## 平坦なデータ：TSVが圧倒的に安い

50件の平坦レコードを9通りに直列化した実測結果だ。`o200k_base` 基準で、pretty JSON（4,128トークン）を0%とした。

| 形式 | 文字数 | o200k トークン | cl100k トークン | pretty JSON比 |
|---|---|---|---|---|
| TSV | 3,742 | **1,568** | 1,663 | −62.0% |
| CSV | 3,742 | 1,650 | 1,650 | −60.0% |
| Markdown表 | 4,766 | 1,897 | 1,897 | −54.0% |
| JSON (compact) | 7,985 | 2,578 | 2,593 | −37.5% |
| key: value 行 | 6,982 | 2,708 | 2,708 | −34.4% |
| YAML | 7,834 | 3,159 | 3,159 | −23.5% |
| TOML | 8,533 | 3,176 | 3,191 | −23.1% |
| JSON (pretty) | 10,986 | 4,128 | 4,143 | 0.0% |
| XML | 13,654 | 4,777 | 4,778 | +15.7% |

![データ形式ごとのLLMトークンコスト棒グラフ — 同じ50件のレコードをTSVで入れるとpretty JSON比62%少ないトークンで済む](../../../assets/blog/llm-token-cost-data-format-experiment.png)

一番目を引くのはpretty JSONとTSVの2.6倍差だ。同じ情報。モデルが受け取る事実は一文字も変わらない。それなのに2スペースのインデント、繰り返すキー名、クォート、波括弧がトークンを2.6倍に膨らませる。XMLはさらに上を行き、閉じタグのおかげで全フィールドを二度書くため、pretty JSONより16%高い。XMLをLLM入力形式に使う選択は、ほぼ確実に損だと私は見る。

CSV・TSV・Markdown表が安い理由は単純だ。**キー名をヘッダ1行だけに書き、50行では値だけ並べるからだ。** JSON系はレコードごとに `"warehouse":` のようなキーを50回繰り返す。フィールドが多く行が長いほど、この繰り返しコストが膨らむ。

1レコードに割って見るともっと明確だ。pretty JSONの4,128トークンを50件で割ると1件あたり約82トークン。TSVはヘッダ行(約18トークン)を除いた1,550トークンを50行で割って1件あたり31トークン。同じ9フィールドを1行に収めて、一方は82トークン、もう一方は31トークンを使う。差の正体はデータではなく、キー名9個 × 50回の繰り返しと、それを包むクォートと波括弧だ。モデルにとって `"category":"books"` と `books` は同じ事実なのに、前者はその事実を伝えるのに3〜4倍のトークンを使う。

## 高いのはキーの繰り返しでなく、ボイラープレートだ

ここで起きやすい誤読を正したい。正確な言い方は「JSONが遅い」ではなく「構造表示文字が高い」だ。compact JSONがpretty JSONより37.5%安いのはキー数が減ったからではない。キーは同じく50回繰り返される。減ったのはインデント空白と改行だけだ。

```text
インデント2スペース "  "  -> 1 token (id 220)
3 commas           ",,,"  -> 1 token
```

`o200k_base` では2スペースのインデントがそれ自体で1トークン(ID 220)を占める。50件のレコードがそれぞれ9フィールドを2スペースずつ字下げすれば、この空白トークンだけで数百個が敷かれる。そこに行ごとの改行、オブジェクトごとの開き/閉じ波括弧が加わる。人の目には可読性でも、モデルの目にはただのコストだ。だから私は「人が直接読むのでなければpretty禁止」を既定値にした。

## ネストデータでは結論が逆転する

ここで止めていたら「とにかくCSV」という誤った教訓を得ていただろう。だから形の違うデータをもう一度測った。注文20件、各注文が顧客オブジェクト(住所までネスト)と可変長の品目配列を持つネスト構造だ。

| 形式 | o200k トークン | pretty JSON比 |
|---|---|---|
| JSON (compact) | **1,538** | −45.7% |
| YAML | 1,958 | −30.9% |
| TOML | 2,021 | −28.7% |
| JSON (pretty) | 2,835 | 0.0% |

CSV・TSV・Markdown表は候補から完全に外れる。可変長の品目配列とネストオブジェクトを2次元の表に押し込む方法がないからだ。そして平坦データで−34%と中位だったcompact JSONが、ネストでは−45.7%で1位になる。YAMLは平坦でもネストでも、インデントコストのせいで思ったほど安くなかった。「YAMLは人にも読みやすくトークンも少ない」という通念は、少なくともこの計測では当たらなかった。

まとめると境界はこうだ。**均一な行なら表系(CSV/TSV/Markdown)、ネストならcompact JSON。** この一行が今日の実験で得た一番使える規則だ。

## エージェントループでこの差は複利で効く

数千トークンは大したことなく見えるが、エージェントは同じコンテキストを毎ターン送り直す。50件のカタログをpretty JSONでシステムプロンプトに固定し、30ターンの会話を回すとする。形式をTSVに変えるだけでターンあたり約2,560トークン(4,128 → 1,568)が消える。30ターンなら76,000トークンだ。コンテキストウィンドウが厳しいモデルではこれが「入るか入らないか」を分け、従量課金のモデルではそのまま入力トークン費用になる。

ここで反論が出るかもしれない。「プロンプトキャッシュを使えば同じコンテキストは安くならないか?」その通りだ。システムプロンプトに固定したカタログならキャッシュヒットで費用は大きく下がる。だがキャッシュはトークン数そのものを減らさない。キャッシュ済みのトークンもコンテキストウィンドウの上限はそのまま食うし、キャッシュは通常短いTTLが過ぎると失効して再び全額で埋め直す。さらにツール結果のように毎ターン中身が変わるデータは、そもそもキャッシュ対象になれない。形式でトークンを削るのはキャッシュと競合する技法ではなく補完する技法だ。両方を入れれば両方とも得をする。

これはMCPツールが大きな結果を返すとき特に効く。[FastMCPで作ったサーバ](/ja/blog/ja/fastmcp-python-mcp-server-build-guide-2026)がDB照会の結果をそのままJSONで返せば、その形式がそのままモデルの入力費用になる。サーバ側で平坦な結果をCSV/TSVに直列化して返す小さな決定一つが、エージェント全体のトークン台帳を変える。

もちろん限界もはっきりある。私はトークン数だけを測り、**モデルが各形式を同じように理解するかは測っていない。** API呼び出しなしで再現できる範囲がここまでだったからだ。直感では、CSVのようにヘッダと値の距離が遠い形式は、フィールドの多いデータでモデルを混乱させ得ると見る。ヘッダが上に一度きりで、30行目の7番目の値が何のフィールドかをモデルは位置で数えねばならないからだ。トークンを惜しんで精度を落とせば元も子もない。だから実適用の前に、トークン削減と応答品質を一度はA/Bで併せて見るべきだ。この検証は次の実験に回す。

## 明日から即適用すること

今日の実測で既定値をこう変えた。

- 平坦なレコード配列をコンテキストに入れるときはCSVかMarkdown表。pretty JSON禁止。
- ネスト構造は `json.dumps(x, separators=(",",":"))` でcompactに。`indent=2` は人がデバッグで見るときだけ。
- LLM入力にXMLは使わない。同じ情報に最もトークンを食う。
- 形式を変えてトークンを大きく削ったら、その形式でモデル精度が保たれるか一度は確認する。

データ形式はコードでほぼ自動で決まる値なので、普段は意識しない。ところがLLMコンテキストに入る瞬間、その無造作な `indent=2` 一つがトークン請求書の半分を作りかねない。自分で測るまでは、私もその大きさを過小評価していた。
