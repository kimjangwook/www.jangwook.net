ローカルでサブエージェントを何個か同時に回すパイプラインを組んでいて、当然だと思っていた前提が一つ引っかかった。クラウドAPIはリクエストを並列に投げれば大抵その分早く終わる。だからローカルのOllamaにもエージェント8個を一気につなげば、GPUが勝手に分けて処理して総時間が減るはずだと漠然と信じていた。

ところがM1のMacBookで実際に8個つないでみると、体感がおかしい。1個回すときと8個回すときで、毎秒出てくるトークンの総量がほぼ同じだった。ただ長くかかるだけ。これが私の勘違いなのか実際の挙動なのか曖昧だったので、憶測をやめてその場で測った。

## 何をどう測ったか

環境はApple M1、16GBの統合メモリ、Ollama 0.30.7だ。モデルはgemma4系の小さな派生（約4GB、`ollama ps`上で100% GPUオフロード）を使った。小さいモデルを選んだのは、同時リクエストを8個まで何度も回しても実験が一度で終わる必要があったからだ。

測り方は単純だ。Pythonの`ThreadPoolExecutor`で同じ個数のリクエストを同時に`/api/generate`へ投げ、リクエストごとに`num_predict`を100に固定して各自ちょうど100トークンずつ生成させた。プロンプトは8種を使い回して互いに変えることで、promptキャッシュが結果を歪めないようにした。同時実行数を1、2、4、8と上げながら二つを見た。

- 集計スループット（aggregate tokens/sec）：バッチ全体が毎秒何トークン吐くか。
- 壁時計時間（wall-clock）：バッチが終わるまで実際に何秒かかるか。

レスポンスJSONの`eval_count`と`eval_duration`をそのまま読み、リクエスト別の実生成速度も併せて記録した。ここで`eval_duration`はキューで待った時間ではなく純粋な生成時間だけを含むので、「順番待ちしていたのか」と「遅く生成されたのか」を切り分けるのに決定的だった。

核心の測定コードはこれで全部だ。再現したければそのままコピーして自分の機器で回せばいい。

```python
import json, time, urllib.request, concurrent.futures, statistics

URL = "http://localhost:11434/api/generate"
MODEL = "gemma4:latest"   # 自分のモデルに差し替え
NUM_PREDICT = 100

def one_request(idx):
    body = json.dumps({
        "model": MODEL,
        "prompt": PROMPTS[idx % len(PROMPTS)],  # 互いに異なるプロンプト8個
        "stream": False,
        "options": {"num_predict": NUM_PREDICT, "seed": idx},
    }).encode()
    req = urllib.request.Request(URL, data=body,
                                headers={"Content-Type": "application/json"})
    with urllib.request.urlopen(req, timeout=600) as r:
        d = json.loads(r.read())
    return d["eval_count"], d["eval_duration"] / 1e9  # トークン数, 純生成秒

def bench(concurrency):
    t0 = time.perf_counter()
    with concurrent.futures.ThreadPoolExecutor(max_workers=concurrency) as ex:
        res = list(ex.map(one_request, range(concurrency)))
    wall = time.perf_counter() - t0
    total_tokens = sum(c for c, _ in res)
    return total_tokens / wall  # 集計スループット(tok/s)
```

`num_predict`を固定するのがポイントだ。これを決めないとリクエストごとに生成長がバラバラになり、スループット比較がぼやける。長さを固定して初めて「同じ仕事を何個同時にさせたとき」の純粋なスループットが出る。

## 既定サーバはリクエストを順番待ちさせる

まず普段使いの既定Ollamaサーバ（ポート11434、環境変数の設定なし）にそのまま投げた。結果の要約はこうだ。

| 同時リクエスト | 壁時計(秒) | 集計スループット(tok/s) | リクエスト別生成速度(tok/s) |
|---|---|---|---|
| 1 | 4.38 | 21.7 | 24.6 |
| 2 | 8.71 | 22.4 | 23.6 |
| 4 | 16.53 | 22.6 | 23.7 |
| 8 | 32.04 | 23.5 | 24.5 |

数字が綺麗すぎて最初はバグかと思った。壁時計時間が同時実行数にきっちり比例する。8個つなぐと1個のときの丁度8倍かかる。一方、集計スループットは22〜23トークン/秒に張り付いてびくともしない。

肝は三列目だ。リクエスト別の生成速度が同時実行数と無関係に常に24トークン/秒あたりを保つ。これが何を意味するかというと、各リクエストは単独のときの最高速度で生成されているということだ。なのに総量が増えず壁時計だけ伸びたのは、リクエストが同時に計算されるのではなく、一度に一つずつ順番に処理されたという意味になる。つまりサーバはリクエストをキューに入れて直列に流した。並列スロットが1だったのだ。

数字で改めて確かめると辻褄が合う。同時8個のとき壁時計は32秒だが、リクエスト1本が100トークンを24トークン/秒で作るのに約4.2秒かかる。4.2秒 × 8 = 33.6秒。観測された32秒とほぼ一致する。八つのリクエストが重ならず順番に処理されたということだ。先に着いたリクエストはすぐ処理されるが、八番目にキューへ並んだリクエストは前の七つが終わるまで純粋に待つだけで、最後に自分の4.2秒を使う。この待ち時間が個々のリクエストの体感遅延をそのまま押し上げる。

この時点で[num_ctxのせいで長い入力の指示が丸ごと切れ落ちた実験](/ja/blog/ja/ollama-num-ctx-silent-truncation-experiment)を思い出した。あのときも「モデルが愚かなのではなく設定の問題」だった。今回も同じ匂いがした。モデルが並列できないのではなく、サーバが並列しないよう設定されているのではという疑いだ。

## なぜ1だったのか — ドキュメントで確認

Ollama公式FAQが答えを持っていた。`OLLAMA_NUM_PARALLEL`がモデル一つが同時に処理する最大リクエスト数を決めるのだが、この値を指定しないと利用可能メモリに応じて4か1が自動選択される（[Ollama FAQ](https://docs.ollama.com/faq)）。そして決定的な一行がもう一つある。並列リクエストはコンテキストサイズを並列数だけ掛け算する。2Kコンテキストに並列4なら実効8Kになり、その分メモリを余計に食う。

つまり16GB M1はメモリが厳しいので、Ollamaが自ら並列を1に下げて設定したのだ。私が怠けたからではなく機器がそう判断した。これはノートPCでローカルモデルを回す人の大半に当てはまる既定値だという意味でもある。

## スロットを4に開けると何が起きるか

仮説を確認するため、二つ目のサーバを別ポート(11500)に`OLLAMA_NUM_PARALLEL=4`で立てた。既存サーバは触っていない。サーバログで`OLLAMA_NUM_PARALLEL:4`が実際に効いたのを確認して同じベンチを回した。

```
model=gemma4(small) num_predict=100  OLLAMA_NUM_PARALLEL=4
concurrency=1  wall=5.15s  aggregate=18.4 tok/s  per_req=21.7 tok/s
concurrency=2  wall=10.57s aggregate=18.5 tok/s  per_req=9.8 tok/s
concurrency=4  wall=11.36s aggregate=33.2 tok/s  per_req=9.9 tok/s
concurrency=8  wall=22.30s aggregate=33.6 tok/s  per_req=10.0 tok/s
```

ここで絵が完全に変わる。同時4個の地点で集計スループットが18トークン/秒から33トークン/秒へ跳ねる。単一ストリーム比で約1.8倍だ。壁時計で見ても同時4個が11.4秒で、既定サーバが同じ4個を処理した16.5秒よりはっきり短い。並列バッチが実際に働いたのだ。

ただし只ではない。リクエスト別の生成速度を見ると同時2個の時点ですでに22から10トークン/秒へ半減している。リクエストが本当に同時に計算されるが、GPU一つを分け合うので各自はその分遅くなる。だから同時2個のときは面白いことに集計スループットが上がらない(18.5のまま)。二つが半分ずつ分け合うので合計が変わらない。利得は4個までスロットを埋め切ったときに初めて出る。

同時8個はどうか。集計スループットが33.6で4個とほぼ同じだ。スロットが4個しかないので残り4個はキューで待つ。だから壁時計は再び伸び(22.3秒)、キューの後ろに並んだリクエストの体感遅延は最大22秒まで開く。スループットはすでに4で天井を打った。

この対比を一枚に描くと下のようになる。左が集計スループット、右が壁時計時間だ。グレーが既定(直列)、青がnum_parallel=4だ。

## で、エージェントを何個つなぐか

この測定が私の[マルチエージェント・オーケストレーション](/ja/blog/ja/multi-agent-orchestration-improvement)設計をすぐ変えた。まとめるとこうだ。

一つ目、既定値では同時実行はスループットを上げない。スロットが1ならエージェントを8個に増やしてもキューが伸びるだけで総時間は8倍になる。「並列に投げたから速いだろう」はローカルでは誤った直感だ。クラウドAPIの感覚をそのまま持ち込んではいけない。

二つ目、スループットが目的なら`OLLAMA_NUM_PARALLEL`を上げるのが正解で、スロットを埋め切ったときだけ利得が出る。スロット4個にリクエスト2個を送れば利得は0だ。バッチをスロット数に合わせてまとめて投げる必要がある。

三つ目、遅延が目的ならむしろ直列がいい。ユーザーが一つの応答を待つ対話型なら、リクエストを一つずつ最高速で終わらせる方が各応答を早く返せる。複数をバッチにまとめると一個あたりの速度が半減するからだ。

私の結論はこうだ。私はバックグラウンドのバッチ作業(文書を複数要約、大量分類)はnum_parallelを上げてまとめて投げ、人が待つ対話経路は直列のままにすることにした。一つの魔法の値で両方を満たす設定はなかった。

## まず自分の機器のスロット数を確認せよ

設計を変える前に、まず自分のサーバが今並列を何個で構えているかを知る必要がある。これは推測するものではなく、すぐ確認できる。サーバを立てたターミナルのログに`server config`が出るので、そこから`OLLAMA_NUM_PARALLEL`の値をそのまま読めばいい。自動選択なら実測で確かめるのが確実だ。上のスクリプトで同時4個を投げてみて、壁時計が単一リクエストの4倍に伸びればスロットは1、ほぼそのままならスロットは4以上だ。

値を変えるにはサーバをその環境変数と一緒に立て直せばいい。

```bash
# 並列スロットを4に開いてサーバ起動
OLLAMA_NUM_PARALLEL=4 ollama serve
```

アプリで自動起動するOllamaなら、システムサービスの環境にこの変数を入れる必要がある。そして変えた後は必ず測り直せ。スロットを増やしたのにモデルがメモリから押し出されるとかえって遅くなることがあるので、設定値だけ見て安心してはいけない。私はこの確認手順を飛ばして「4にしたから大丈夫だろう」とやり、大きいモデルでスロットが開かないのを後から気づいたことがある。

## この測定の限界

正直に押さえておく。これはM1 16GBに小さいgemma4を載せた特定条件の数字だ。メモリが潤沢な機器や専用GPUがある環境なら既定のnum_parallelは4になるだろうし、大きいモデルはスロットあたりのKVキャッシュが大きくてそもそも並列スロットを数個しか立てられない。並列スロットがコンテキストメモリを掛け算する点のせいで、長いコンテキストを使うエージェントほど同時実行を上げにくい。だから[単一リクエストのprefillコストを測った記事](/ja/blog/ja/local-llm-prefill-generation-latency-experiment)で見た長い入力の重さが、ここで再び足を引っ張る。

もう一つ。集計スループット1.8倍は、このハードウェアでGPUが遊んでいた余裕があったという意味でもある。単一ストリームがGPUをすでに100%使っていたら、並列にまとめても利得はほぼ無かっただろう。利得の大きさは機器ごとに違うので、結論は一つだ。設定を変えたら必ず自分の機器で測り直せ。30行のスクリプトで十分だ。

次は大きいモデル(gemma4:12b)で同じ実験を回して、モデルが大きくなるとき並列スロットが何個まで耐えるかと、その地点でGPUが飽和する境界を測るつもりだ。ローカルでエージェントを複数回す以上、この境界値がそのまま私のパイプラインの実質上限になる。
