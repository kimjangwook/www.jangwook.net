先月、[temperatureとseedで出力の再現性を測った記事](/ja/blog/ja/llm-determinism-temperature-seed-experiment)で、私は一段落を自信たっぷりに間違えて書いた。gemma4:12b-it-qatが `eval_count` は上がるのに `content` が空文字で返るのを見て、「トークンが画面テキストにマッピングされないパッケージングの問題」と断定し、そのモデルを再現性の表から外してしまった。

そうではなかった。今回、別の実験を準備していて同じ空応答に再び出くわし、今度は応答JSONを最後まで開いた。`message` の中に `content` のほかに `thinking` というフィールドがもう一つあった。gemma4:12bは推論(reasoning)モデルだったのだ。空応答はバグではなく、私が `num_predict` を低くしすぎて生成予算がすべて推論チャネルに吸われ、肝心の答えにトークンが一つも残らなかった結果だった。診断ごと間違えていたわけだ。

誤診を正すと、もっと知りたいことが出てきた。この推論というやつ、元が取れるのか。同じ答えを出すのに23倍遅く84倍のトークンを使うなら、エージェントで点けるべきか消すべきか。だから実測した。

## 空応答の真犯人はthinkingフィールドだった

まず再現。最も単純な算数を二つのモードで投げた。Ollamaの `/api/chat` では、リクエスト本文の最上位に `think` ブール値を置くと推論を点けたり消したりできる(`options` の中ではなく `model`・`messages` と同じ階層だ)。

```python
import json, urllib.request
def chat(msg, think, num_predict=512):
    body = {
        "model": "gemma4:12b-it-qat",
        "messages": [{"role": "user", "content": msg}],
        "stream": False,
        "think": think,                       # ← ここ。optionsの外
        "options": {"temperature": 0, "seed": 7,
                    "num_ctx": 2048, "num_predict": num_predict},
    }
    req = urllib.request.Request("http://localhost:11434/api/chat",
        data=json.dumps(body).encode(),
        headers={"Content-Type": "application/json"})
    r = json.load(urllib.request.urlopen(req, timeout=300))
    m = r["message"]
    return m.get("content", ""), m.get("thinking") or "", r.get("eval_count")
```

「40ドルが20%引き後の価格のとき、元の価格は？」を投げた結果だ。

| モード | 時間 | 生成トークン | thinking文字数 | 答え |
|--------|------|-------------|----------------|------|
| `think=true` | 37.0秒 | 252 | 551 | 50(正解) |
| `think=false` | 1.6秒 | 3 | 0 | 50(正解) |

どちらも正確に同じ答えを出した。推論を点けた側は23倍遅く、84倍のトークンを燃やした。前回 `num_predict` を24にしたとき、この252個の推論トークンが切られ、答え「50」が生成される前に途切れたのだ。空の `content` はそうして出た。モデルのせいでも、パッケージングのせいでもなく、私の設定のせいだった。[num_ctxのせいで長い入力の指示が丸ごと切れていた前回の実験](/ja/blog/ja/ollama-num-ctx-silent-truncation-experiment)と全く同じ教訓だ。モデルが愚かに見えるとき、犯人はたいてい私のオプションである。

## そこで質問を変えた。推論はどこで元を取るのか

「同じ答えなら切る方がいい」は早すぎる結論だ。算数一問では推論が損に見えるが、推論モデルが存在する理由は、速い直感が外れる問題でもう一度考え直して答えを正すからだ。その「直感が外れる問題」の教科書が認知反応テスト(Cognitive Reflection Test, CRT)だ。シェーン・フレデリックが2005年の論文で示した三問(バットとボール、機械とウィジェット、睡蓮の葉)は、頭に最初に浮かぶ答えがほぼ必ず誤りになるよう設計されている。

そこで仮説を立てた。**推論を点ければCRTの罠で精度が上がるはずだ。** 直感を外すよう作られた問題なのだから、じっくり考えるモードはここで光るべきだ。

13問を難易度3層で組んだ。すべて答えが一つに定まり、「数字だけ答えよ」のように採点可能な形式を求めた。

| 層 | 狙い | 例 |
|----|------|-----|
| 易(A1〜A4) | 単純照会・暗算 | 日本の首都、7×8、一週間は何日 |
| 中(B1〜B4) | 多段階の文章題 | 20%引きの逆算、60km/45分→時速、りんご12個の問題 |
| 難(C1〜C5) | CRTの罠+即席手順 | バットとボール、機械とウィジェット、睡蓮の葉、6つの数の並べ替え、"strawberry"のrの数 |

各問を `think=false` で一度、`think=true` で一度回した。temperature=0、seed固定なので同じモードは再現する。全呼び出しの時間・生成トークン数・正誤を記録し、推論トレースはCRT問題二つ(バットとボール、睡蓮の葉)について丸ごと保存した。

## 結果。正解1個を19倍の時間で買った

26回の呼び出し(13問×2モード)の要約だ。

| 指標 | thinking OFF | thinking ON | 倍率 |
|------|-------------|------------|------|
| 正解数 | 12 / 13 | 13 / 13 | +1 |
| 平均応答時間 | 1.4秒 | 28.3秒 | 20倍 |
| 問題あたり平均出力トークン | 3 | 189 | 63倍 |
| セット全体の総出力トークン | 36 | 2,454 | 68倍 |
| セット全体の総所要時間 | 19秒 | 368秒 | 19倍 |

推論を点けて得たのは正解1個だ。その1個のためにセット全体で出力トークンを68倍、実時間を19倍多く使った。上のヒーローチャートの青い棒(OFF)はほぼ見えない。問題あたり1〜6トークンだからだ。オレンジの棒(ON)は46トークンから399トークンまで伸びる。

![精度とコストの要約。正解1個の差に出力トークンは68倍](../../../assets/blog/local-llm-reasoning-mode-token-cost-experiment/accuracy-vs-cost.png)

最もトークンを燃やしたのは機械とウィジェット(C2)で、推論ONで399トークン、59秒だった。同じ問題を推論OFFは2トークン、1.4秒で解いた。しかも正解で。

## 推論が実際に救ったのはCRT問題ではなかった

ここで仮説が崩れた。推論を点けて初めて正解した問題は一つ、C4だけ。ところがC4はCRTの罠ではない。「17, 3, 29, 8, 21, 14 を降順に並べて3番目に大きい数を答えよ」という即席の手順問題だ。推論OFFは21と答え(2位を3位と取り違えた)、推論ONは29・21・17と辿って17を当てた。

肝心の、光ると見込んだCRTの罠三問は推論を切っても全て正解した。

| 問題 | 直感的な誤答 | OFF | ON |
|------|-------------|-----|-----|
| C1 バットとボール(ボールは何セント？) | 10 | 5 ✓ | 5 ✓ |
| C2 機械とウィジェット(何分？) | 100 | 5 ✓ | 5 ✓ |
| C3 睡蓮の葉(半分は何日目？) | 24 | 47 ✓ | 47 ✓ |

なぜこうなるのか。正直、私に見えていない変数がある。この三問は認知科学の教科書とLLM評価論文に数えきれないほど登場した結果、答えごと学習データに刻まれている公算が大きい。gemma4:12bはこれを「推論」したのではなく「記憶」して即答したのかもしれない。逆にC4の6つの数はその場で私が選んだので暗記しようがない。暗記では解けない即席手順でのみ推論が実際に働いた、という解釈が私のデータに最もよく合う。

もう一つ意外だったのがC5だ。「strawberryにrは何個」はトークナイザーが単語を丸ごと飲み込むためLLMが伝統的に外してきた罠だが、gemma4:12bは推論を切っても3と即答した。これも同じ質問が学習分布に多すぎて固着した結果に見える。つまり「有名なLLMの罠」という分類自体、もはや推論の価値を測る物差しとして鈍ってしまった。モデルがその罠を罠として覚えてしまったからだ。

だから私はこう見る。**推論モードの価値は「難しい問題」ではなく「モデルが見たことのない多段階の手順」から出る。** 有名なひっかけ問題で推論モデルを評価すると、点けても消しても正解して差が見えない。差が開くのは、モデルが初めて見る具体的なデータを自分で捌かねばならないときだ。

## 推論トレースを開くと見えるもの

推論ONがバットとボール(C1)で生成したthinkingをそのまま写す。

```
*   Total cost of bat + ball = $1.10.
*   Let x be the price of the ball.
*   The bat costs x + 1.00.
*   Equation: x + (x + 1.00) = 1.10
*   2x = 0.10  ->  x = 0.05
*   Convert to cents: 0.05 x 100 = 5.
*   検算: Ball 5 + Bat 105 = 110 cents. Correct.
ANSWER: 5
```

式を立て、検算までする。教科書通り正しい解法だ。問題は、推論OFFも同じく5と即答したことだ。297トークンを費やしたこの綺麗な導出が、結果だけ見れば2トークンの即答と同点だ。過程が見事でも元を取ったわけではない。睡蓮の葉(C3)のトレースも同様で、「毎日倍だから満杯の一日前が半分」という核心を正確に突く。だがこの洞察もOFFがすでに持っていた(47即答)。推論は答えを変えられず、ただその答えに至る経路を見せただけだ。経路をログに残す用途(評価・デバッグ・監査)ならそれ自体が価値だ。だが最終答えだけ使うパイプラインなら297トークンはただのコストである。

## エージェントに適用しながら変えたこと

この実測のあと、私のローカルエージェント設定で三つを変えた。

第一に、**照会・分類・フォーマット変換の段階は `think=false` を既定値に固定した。** ルーティング(「この要求はどのツールへ？」)、短い抽出、JSON整形といった作業は直感が外れる余地がほぼない。ここで推論を点けるのは段階ごとに20倍の遅延と60倍のトークンを献上することだ。エージェントはこうした軽い段階を何十回も通るので累積損失が大きい。[エージェントを一度回すときトークンがどこから漏れるかを分解した記事](/ja/blog/ja/ai-agent-cost-reality)で見たように、コストは大きな一発ではなく小さな段階の反復から漏れる。

数字で押さえれば判断が楽になる。ルーティング・抽出の段階が一要求に10回回るとしよう。推論OFFなら一段階1.4秒、10段階で14秒。推論ONなら一段階28秒、10段階で280秒。ユーザーは4分以上も空の画面を見る。その代償に得る精度向上は、少なくとも直感が外れないこれらの段階ではほぼ0だ。ローカルモデルはトークンあたり現金が出なくても、時間と電力は実在のコストなのでこの計算はそのまま効く。

第二に、**推論は「モデルが初めて見る多段階の計算」にだけ点ける。** ユーザーデータをその場で集計したり、複数の制約を同時に満たしたり、中間状態を持ち越したりする段階だ。C4がまさにその形だった。有名なパズルを解くから点けるのではなく、暗記された答えのない手順を回すときに点ける。

第三に、**推論を点ける段階には `num_predict` を多めに与える。** 最初に食らった空応答が、まさにこれを怠った結果だ。推論チャネルが189トークン前後を先に食うので、答えまで出るにはその上に余裕が要る。点けるなら予算も一緒に上げる。[出力の再現性を扱った記事](/ja/blog/ja/llm-determinism-temperature-seed-experiment)の推奨設定にもこの項目を足した。

## 一モデル、13問の実験の限界

正直に書く。これはgemma4:12b一モデル、13問、一問一回の計測だ。統計というより一人がノートPCで一日触った記録に近い。CRT問題が学習データにあったかは私に確かめる術がなく、「公算が大きい」までしか言えない。より大きな推論モデルやより難しいベンチマークでは、推論の利得は確実にもっと大きく出るだろう。この記事の数字を「推論は無用」と読まれては困る。私の結論はそれではなく、「推論はタダではなく、点ける場所を選ばねばならない」だ。

元々やろうとしたのは長いコンテキストで情報位置による検索精度(いわゆるlost-in-the-middle)を測ることだったが、1.5kトークンのプリフィルに26秒かかり、単発実行の中で終えられなかった。それは次の機会に回す。代わりに今日は空応答の真因を突き止め、推論の元を測れたので十分な商売だった。少なくとも前回の記事の誤診を一つ正したのだから。
