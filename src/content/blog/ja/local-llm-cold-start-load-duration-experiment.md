MacBookでローカルエージェントを数日回している。作業の合間に少し別のことをして、同じエージェントを呼び直すと、最初の応答だけがやけに重い。二回目、三回目はまともなのに、最初の一回だけがもたつく。昨日[prefillとgenerationのコストを分解した記事](/ja/blog/ja/local-llm-prefill-generation-latency-experiment)を書いたとき、計測前にモデルを一度ウォームして「モデル読み込み時間(load_duration)が計測に混ざらないようにした」と書いた。その一文を書きながら、少し後ろめたかった。わざと外したその項目こそ、普段いちばん体感している遅延だったからだ。

そこで今日は、昨日外したそのコストを正面から測った。モデルがメモリに乗るまでの時間、いわゆるコールドスタートだ。

## load_durationという、普段は見えない項目

Ollamaの`/api/generate`は応答ごとにタイムスタンプの束を一緒に返す。昨日はそのうち`prompt_eval_duration`(prefill)と`eval_duration`(generation)だけ見たが、先頭に`load_duration`という項目がもう一つある。名前のとおり、モデルを読み込むのに使った時間だ。

これが普段見えにくいのには理由がある。同じモデルを連続で呼ぶと、二回目からはモデルがすでにメモリに乗っているので`load_duration`がほぼ0に出る。そして少し使わないとOllamaがモデルをメモリから降ろし(デフォルト5分)、次の呼び出しで読み込みコストが復活する。私が「少し離れて呼び直すと遅い」と感じた正体が、まさにこれだった。

計測方法は単純にした。読み込み時間だけ切り出すために、プロンプトは`Reply with the single word: ok`の一行、`num_predict`は8に絞って生成をほぼ0に収束させた。コールド状態を強制するには、呼び出しの直前に`ollama stop <model>`でモデルを降ろせばいい。そのうえで最初の呼び出しの`load_duration`を読めば、それがコールドスタートだ。

```python
def gen(model, keep_alive="5m"):
    body = json.dumps({
        "model": model, "prompt": "Reply with the single word: ok",
        "stream": False, "keep_alive": keep_alive,
        "options": {"num_predict": 8, "temperature": 0}
    }).encode()
    req = urllib.request.Request(OLLAMA, data=body,
        headers={"Content-Type": "application/json"})
    with urllib.request.urlopen(req, timeout=600) as r:
        d = json.loads(r.read())
    return d["load_duration"] / 1e6  # ナノ秒 -> ミリ秒
```

自分で確かめたいなら、curl一行で済む。モデルを一度降ろして呼び出し、`load_duration`だけ取り出す。

```bash
ollama stop gemma4:12b-it-qat
curl -s http://localhost:11434/api/generate -d '{
  "model": "gemma4:12b-it-qat", "prompt": "ok", "stream": false
}' | python3 -c 'import sys,json; print(json.load(sys.stdin)["load_duration"]/1e9, "秒")'
```

値はナノ秒で来るので1e9で割って秒に直す。この一行をモデルを変えながら回すと、下の表が自分のハードでどう変わるかすぐ掴める。

この計測を信じていい理由が一つある。Ollamaは`load_duration`を`prompt_eval_duration`や`eval_duration`とは別に返す。つまり読み込み時間がprefillやgenerationの数字に混ざらない。応答の`total_duration`はおおむねこの三つの合計に近く落ちたので、読み込みだけきれいに切り出せた。昨日は真ん中の二つを見て、今日は先頭の一つだけに絞ったわけだ。

## モデルサイズ別にコールドスタートを測った

手元に落としてあるGemma 4系のモデル四つをサイズ順に並べた。各モデルを`ollama stop`で降ろしてからコールド状態で三回、そしてモデルが乗ったウォーム状態で一回呼び出した。秒単位で整理するとこうだ。

| モデル | ディスクサイズ | コールド #1 | コールド #3 | ウォーム |
|---|---|---|---|---|
| melavisions/gemma4 | 2.0 GB | 3.33秒 | 1.55秒 | 0.20秒 |
| yinw1590/gemma4-e2b | 3.1 GB | 3.57秒 | 1.79秒 | 0.38秒 |
| gemma4:12b-it-qat | 7.2 GB | 9.00秒 | 2.82秒 | 0.37秒 |
| gemma4:e4b | 9.6 GB | 9.71秒 | 3.86秒 | 0.37秒 |

![モデルサイズ別コールドスタートload_duration比較](../../../assets/blog/local-llm-cold-start-load-duration-experiment/hero.png)

大きな流れは予想どおりだ。モデルが大きいほど読み込みが長い。9.6GBモデルの最初のコールドは9.7秒、同じ呼び出しがウォーム状態では0.37秒だった。26倍の差だ。要するに、7.2GBのローカルチャットボットを立てて5分以上放置し、また話しかけると、答えが出る前に数秒をただ捨てるということだ。

目を引くのはウォームの値だ。モデルが2GBでも9.6GBでも、ウォーム状態の`load_duration`は0.2〜0.4秒でほぼ横ばいだった。サイズに依存しない。私の理解では、これは実際に重みを読み直しているのではなく、Ollamaが「このモデルまだ乗ってるな」を確認するkeep_alive点検程度のオーバーヘッドだ。本当の読み込みではないのでサイズを引かない。正確に何の作業かまでは断定しない。ただ、ウォーム状態の0.4秒は「実質読み込みコストなし」と見ていい、というのが計測から見た結論だ。

## 「コールド」なのに、なぜ#1と#3が二倍も違うのか

表をもう一度見ると、おかしな点が一つある。確かに毎回`ollama stop`でモデルを降ろして測り直したのに、コールド#1がコールド#3より二倍近く遅い。7.2GBモデルは9.00秒から2.82秒へ、9.6GBモデルは9.71秒から3.86秒へ落ちた。同じ「コールド」と呼んだのに値が違う。

ここでしばらく悩んだ。最初は計測バグを疑ったが、答えはOSのページキャッシュだった。`ollama stop`はOllamaプロセスのメモリからモデルを降ろすだけで、OSは一度読んだモデルファイルをRAMのページキャッシュにそのまま抱えている。だからコールド#2、#3はディスクではなくRAMからファイルを読み直す。ディスクI/Oが丸ごと抜けるので速くなる。

これがなぜ重要かというと、私たちがよく言う「コールドスタート」が実は二種類だという意味だからだ。

- 本当のコールド: 再起動直後やメモリ圧迫でキャッシュが空になった状態。重みをディスクから初めて読む。コールド#1にあたる。
- キャッシュ済みコールド: モデルはメモリから降りたが、ファイルはページキャッシュに残った状態。コールド#3にあたる。

ベンチマーク時にこの二つを区別しないと、二回目以降はこっそりキャッシュ済みの値が出て、「コールドスタートは思ったより速い」という楽観的な結論になる。実際の本番サーバーは再起動もするし、複数のモデルを交互に立てるとページキャッシュが押し出される。だからSLAやコールドスタート予算を決めるときは、コールド#3ではなくコールド#1、つまり再起動直後の最悪値を基準にするのが安全だ。私もこれを知らずに一度だけ測っていたら、7.2GBモデルのコールドスタートを「2.8秒」と書いていただろうが、実際の最悪は9秒だった。

興味深いのは、小さいモデルではこの差がずっと小さいことだ。2.0GBモデルはコールド#1が3.33秒、#3が1.55秒で約1.8秒差なのに、9.6GBモデルは9.71秒と3.86秒で6秒近く開いた。ディスクから読むバイトが多いほど、ページキャッシュが節約してくれる時間も大きいということだ。大きいモデルほど「再起動直後の最初のユーザー」が背負うペナルティが急に大きくなる。ローカルで13B級以上をサービスするつもりなら、このキャッシュ依存を運用変数として真剣に見るべきだ。

## keep_alive一つで費用が分かれる

コールドスタートを避けるいちばん直接的なつまみが`keep_alive`だ。モデルをメモリにどれだけ留めるかを決める値だ。これを両極端に置いて、同じ7.2GBモデルに三回ずつリクエストした。

| リクエスト | keep_alive="0" (毎回アンロード) | keep_alive="10m" (ウォーム維持) |
|---|---|---|
| リクエスト #1 | 7.10秒 | 2.56秒 |
| リクエスト #2 | 2.55秒 | 0.38秒 |
| リクエスト #3 | 2.55秒 | 0.38秒 |

![keep_alive設定によるload_duration比較](../../../assets/blog/local-llm-cold-start-load-duration-experiment/keepalive.png)

差が鮮明だ。`keep_alive="0"`はリクエストを処理した直後にモデルをすぐ降ろす。だから全リクエストがコールドだ。毎リクエストが2.5秒以上を読み込みに先払いする。`ollama ps`で確認すると、リクエストの合間にモデルがメモリにない。

逆に`keep_alive="10m"`は最初のリクエストだけコールドの値(2.56秒)を払い、以降のリクエストは0.38秒に落ちる。コールドスタートを最初のリクエストに一度だけまとめ、残りはウォームで処理するわけだ。リクエスト#1のkeep_alive=0の値が7.1秒と特に跳ねたのは、その時点でページキャッシュも空だった本当のコールドだからだ。前の節で見たあの効果が、ここでもそのまま出ている。

コマンドラインでも`OLLAMA_KEEP_ALIVE`環境変数やAPIの`keep_alive`フィールドで同じものを調整する。`-1`にすれば無期限でメモリに留める。

## では、ローカルエージェントをどう立てるか

測ってみると、普段ぼんやりしていた運用判断がいくつかはっきりした。

一つ目、対話やエージェント用途なら`keep_alive`を十分に与えるべきだ。ユーザーが一度話しかけるたびにモデルが降りていると、毎ターンがコールドスタートだ。7.2GBモデルでターンごとに2.5秒ずつ先払いするのは、会話体験を壊す。メモリが許す限り`keep_alive`を長くするか`-1`で固定するのがいい。これは私の[OllamaとFastAPIで本番サービングを組んだ記事](/ja/blog/ja/ollama-fastapi-production-deployment-guide-2026)で扱ったデプロイ構成にそのまま乗せられる設定だ。

二つ目、サーバーを立てるときにモデルを先に一度ウォームしておくといい。起動スクリプトでダミープロンプトを一度呼んでコールドスタートを先に払っておけば、最初の実ユーザーが9秒を背負うことがない。最初のリクエストにコールドコストをまとめるのは避けられないが、その最初のリクエストが本物のユーザーである必要はない。

三つ目、複数のモデルを交互に使うルーティングは思ったより高い。リクエストごとに違うモデルを呼ぶとその都度読み込みが再び起き、メモリが足りないと互いのページキャッシュを押し出して本当のコールド(#1水準)へ落ちる。モデル4つを回すルーターを書くなら、読み込みコスト × 切り替え回数を先に計算しておくといい。

四つ目、推論速度のベンチマークは必ずウォーム後に測るべきだ。昨日の記事で私が計測前にウォームした理由がこれだ。コールド状態で一度だけ測ると`load_duration`がprefill・generationの上に9秒ずつ乗って、モデルが遅いのか読み込みが遅いのか区別がつかない。[出力の再現性を測った実験](/ja/blog/ja/llm-determinism-temperature-seed-experiment)でも同じ原則が効いた。測りたい項目以外の変数は先に固定する。

五つ目、メモリと応答性は引き換えの関係だ。`keep_alive`を長くすれば最初のリクエスト以外のコールドスタートは消えるが、そのモデルがRAMを占有し続ける。9.6GBモデルを無期限で立てておくと、その分ほかの作業が使えるメモリが減り、別のモデルを立てた瞬間にページキャッシュが押し出されてコールドが復活する。だから私は「どのモデルを常に立てておくか」を先に決め、よく使う一つ二つだけ`keep_alive`を長く、残りは短くと分けた。すべてのモデルをウォームで抱えておくのは、メモリが支える時だけの贅沢だ。コールドスタートを0にしようと無闇に`keep_alive=-1`を打ち込むと、次のモデル読み込みでより大きなコールドになって返ってくる。

## 限界と、まだ分からないこと

正直に境界を引いておこう。これはMacBook(Apple Silicon、統合メモリ)一台で測った値だ。CUDA GPUの載ったサーバーは、読み込み経路がディスクからシステムRAMを経てVRAMへコピーされる段が増えるので、絶対値は違って出るはずだ。私の数字をそのまま別のハードに移さないこと。ただ「コールドはサイズを引き、ウォームは引かない」「ページキャッシュの有無でコールドが二種類に分かれる」「keep_aliveが最初のリクエスト以外の費用を決める」という構造的結論は、ハードが変わっても保たれると見ている。

もう一つ、`load_duration`が正確にどんな作業の合計かはOllama内部を深く掘っていないので断定しない。ファイル読み込みだけでなく、演算グラフの構成のような初期化も混ざっているかもしれない。私が観測できるのはAPIが返すその数字で、その数字がモデルサイズ・ページキャッシュ・keep_aliveにどう反応するかまでが今日の計測の範囲だ。ウォーム状態でも0.37秒が出る正体も推定しただけで確定はできていない。

最後に、ページキャッシュの挙動は空きRAM量に左右される。メモリのきついサーバーなら、コールド#2、#3でさえキャッシュがすぐ押し出されてコールド#1のように遅くなりうる。私の計測はRAMが潤沢な状態での楽観的なケースに近い。次はメモリ圧迫を人為的にかけて、キャッシュがどれだけ持つか測ってみたい。コールドスタートは一度の計測で終わる話ではなく、環境ごとに測り直す種類のコストだ。
