我输入了一句英语。「I refactored the agent loop to cut token usage.」用OpenAI的o200k_base是12个token。再把同样的意思换成韩语：「토큰 사용량을 줄이려고 에이전트 루프를 다시 짰다.」20个token。字符数韩语还不到一半，token却是1.7倍。

我想知道这是一句话的偶然，还是铺在我整个博客底下的结构性成本。恰好手上有绝佳的实验材料。这个博客每篇文章都出四个版本：韩语、日语、英语、中文。也就是说，语义相同的文档在四种语言里有285组。撇开翻译质量的争论，我手边已经有了一个能干净测量"内容相同、只换语言时token增加多少"的数据集。

于是我把285篇×4语言全部用三种真实分词器分了词。先说结论：非英语的token税是真实的，比我想的更大，而且换模型会改变税率。

## 为什么我得自己测

"韩语比英语更费token"是社区里常听到的说法。可一旦问"具体几倍"，答案就五花八门。有人说2倍，有人说1.2倍。当然会这样。测的文本不同，分词器不同，是否混入代码块也不同。

我需要的不是飘着的数字，而是"我的博客，用我实际使用的模型跑一遍"时落下来的成本。翻译、摘要、嵌入、RAG上下文注入，我的自动化流水线几乎每一步都按token计费。如果韩语比英语贵1.4倍，那就是直接印在我每月账单上的数字。

我把测量目标定为三种分词器。最新的OpenAI系o200k_base(GPT-4o/GPT-5代)、旧的cl100k_base(GPT-4/3.5代)，以及Claude分词器。三者都属BPE系，但学习词汇的数据不同，对非英语的处理也就不同。我日常用的模型大致都落在这三条线里。

## 怎么测的(只剥掉frontmatter，正文整段送进去)

没用什么花哨工具。tiktoken离线就能直接跑，Claude分词器则从Hugging Face的`Xenova/claude-tokenizer`加载。每个Markdown文件只切掉YAML frontmatter，把正文整段(含代码块)原样分词。真正进入LLM的就是整段正文，所以我故意没有清洗。

```python
import tiktoken
from transformers import AutoTokenizer

o200k  = tiktoken.get_encoding('o200k_base')   # GPT-4o / GPT-5 代
cl100k = tiktoken.get_encoding('cl100k_base')  # GPT-4 / 3.5 代
claude = AutoTokenizer.from_pretrained('Xenova/claude-tokenizer')

def strip_frontmatter(text):
    if text.startswith('---'):
        end = text.find('\n---', 3)
        if end != -1:
            text = text[end + 4:]
    return text.strip()

# 仅统计四语言都有同一slug的文章(285组)
for lang in ['ko', 'ja', 'en', 'zh']:
    body = strip_frontmatter(open(path(lang, slug)).read())
    o200k_tokens  = len(o200k.encode(body))
    cl100k_tokens = len(cl100k.encode(body))
    claude_tokens = len(claude.encode(body))
```

我只把四种语言都齐了同名文件的285篇纳入统计。这样就不会混入"有英语版却没有韩语版"这类样本偏差。这是对同一篇文章四个版本的1:1:1:1比较。

## 数字说了什么

285篇全部相加的结果。单位是token。

| 语言 | o200k(最新) | cl100k(旧) | Claude | 字符数 |
|------|------------:|-----------:|-------:|-------:|
| 英语(en) | 908,938 | 915,128 | 1,003,948 | 3,859,685 |
| 中文(zh) | 1,045,977 | 1,267,007 | 1,340,943 | 2,493,687 |
| 日语(ja) | 1,217,284 | 1,502,403 | 1,579,075 | 2,584,255 |
| 韩语(ko) | 1,256,718 | 1,668,007 | 1,882,800 | 3,076,489 |

把英语设为1.0，非英语的token税一眼就看清了。

| 语言 | o200k倍率 | cl100k倍率 | Claude倍率 |
|------|----------:|-----------:|-----------:|
| 英语 | 1.00 | 1.00 | 1.00 |
| 中文 | 1.15 | 1.39 | 1.34 |
| 日语 | 1.34 | 1.64 | 1.57 |
| 韩语 | 1.38 | 1.82 | 1.88 |

即便用最新分词器(o200k)，韩语也是英语的1.38倍，日语1.34倍。换到旧分词器(cl100k)，韩语拉大到1.82倍。在Claude分词器上，韩语以1.88倍成为三者中最贵的。

我觉得最有意思的是，字符数和token数是各走各的。英语版按字符数最长，386万字。韩语版更短，308万字。可token韩语却比英语多38%。字少，token反而多。按每字符token(tok/char)看：英语0.235，韩语0.408，中文0.419，日语0.471。英语BPE把常用词整个塞进一个token，而谚文、假名、汉字没被这样学习，于是token被切得很碎。

## 韩语为什么更费token，我把「에이전트」切开看了

我想亲眼看到原因，于是把一个词送进分词器，解码它是怎么切的。

```python
w = "에이전트"   # = agent
[o200k.decode([t]) for t in o200k.encode(w)]
# ['에', '이', '전', '트']   → 4个token
```

英语「agent」是1个token。韩语「에이전트」按音节切成4个token。cl100k结果相同。英语里一个词约等于一个token，韩语里一个字(音节块)更接近一个token。这一个词上的4:1差距，累加到整篇文章就成了1.4倍。

把一句短句用三种分词器同时看，会更清楚。所有版本都意为「I refactored the agent loop to cut token usage.」。

| 语言 | 字符数 | o200k | cl100k | Claude |
|------|------:|------:|-------:|-------:|
| 英语 | 47 | 12 | 12 | 12 |
| 中文 | 19 | 14 | 24 | 21 |
| 日语 | 30 | 22 | 28 | 27 |
| 韩语 | 28 | 20 | 31 | 32 |

英语三者都是12。一到非英语，分词器之间就分叉了。同一句韩语在o200k是20，在Claude是32。1.6倍的差距，仅由一个模型选择决定。

日语有它自己的纹理。按每字符token，日语以0.471为三者最高。汉字、平假名、片假名混在一句里，片假名外来词(「エージェント」)还按音节细切。可按整篇文档总量看，韩语反而比日语token多。日语用一个汉字压进很多含义，文档更短；韩语文档本身更长，所以即便每字符效率比日语好，总量上也会反超。效率和总量并不指向同一方向。

## 分词器的代际更替，实质上是给非英语打了个折

这是这次实验里最意外的发现。我算了一下，当分词器从cl100k换到o200k时，各语言的token各减少了多少。

- 英语：减少0.7%
- 中文：减少17.4%
- 日语：减少19.0%
- 韩语：减少24.7%

对英语用户来说，换分词器几乎不改变token数。0.7%属于误差范围。可韩语在同一批文章上便宜了四分之一。OpenAI为o200k扩充词汇，实质上做的事，是悄悄给非英语发了一张折扣。英语本就接近最优，没什么可再榨的了。

![从cl100k换到o200k时的token减少率，折扣集中在非英语](../../../assets/blog/multilingual-llm-token-tax-experiment/tokenizer-discount.png)

这会直接影响实务决策。若是大量跑CJK文本的工作负载，选模型时别只看基准分，也要看分词器代际。同样的价目表，非英语文本上实际账单由分词器决定。我之前[测量数据格式对token费用影响](/zh/blog/zh/llm-token-cost-data-format-experiment)时也是同一教训。模型价目表只是起点，真正的成本取决于输入如何变成token。

## 每篇文章的税不同，不能只信平均

只看总量很整齐，但拆到每篇，方差相当大。我对285篇各自算出韩语对英语的token倍率，分别取中位数和平均(基于o200k)。

| 语言 | 单篇倍率中位数 | 单篇倍率平均 |
|------|--------------:|------------:|
| 中文 | 1.10 | 1.14 |
| 日语 | 1.41 | 1.40 |
| 韩语 | 1.31 | 1.40 |

韩语中位数1.31，平均却是1.40。平均高于中位数，说明少数特别费token的文章把均值拉高了。仔细看，那些文章韩语散文占比高、英语代码和专有名词少。反过来，代码块多的教程类文章倍率在1.1左右，因为代码本就是英语，四语版本里分词几乎一致。

一条实务教训：用语料平均倍率去估单篇成本，在代码少、散文多的文章上会偏得很远。若是成本敏感的单次任务，就直接对那段文本分词。

## 所以我的账单上会出现什么

这个博客从英语初稿起步，铺成另外三种语言，再把整个语料嵌入用于相关文章推荐和搜索。几乎每一步都按token计费。

按整个语料看，285篇英语版在o200k上约91万token。四语全加约443万token。相比只运营英语，多语言不是简单的4倍，非英语税会叠加上去。只看三个译本(ko+ja+zh)约352万token，若按三份英语(273万)估算，就少算了约29%。

这里暴露了我常犯的错误：用英语token数去估韩语成本。这在新模型上偏28%，在旧模型上几乎漏掉一半。用英语报价、按韩语收费，差额每月累积。

具体勾画一下。设想把一篇新文章从英语初稿译成韩语。输入是英语原文(约3,200个token)，输出是韩语译文。而韩语输出在内容相同的情况下要多1.38倍token。输出token单价通常比输入贵，所以非英语税落在最贵的一侧。再译成日语、中文，这笔税就分别叠加三次。在我一篇出四语的运营结构里，非英语输出token在总发布成本中占的块，远大于英语单篇所占的份额。当年我把多语言一路[扩展到中文](/zh/blog/zh/adding-chinese-support)、文章数翻到四倍时，成本不是正好4倍而是更高的原因，就在这里。

为削减token税，我正在改的几件事。反复推同一上下文的推荐流水线，我[启用了提示缓存](/zh/blog/zh/claude-api-prompt-caching-cost-optimization-guide)，防止非英语税在每次调用上重复附加。RAG分块按真实token数切，而不是字符数。把每字符token的数值反过来用，就能得出分块大小。设嵌入模型上下文为512个token，英语按0.235 tok/char能装约2,170字，韩语按0.408 tok/char在约1,250字就触到同一上限。把韩语按和英语一样的"1,000字"切，分块里的token会多出1.7倍，悄悄超出上下文窗口或嵌入上限而被截断。我在[整理韩语RAG嵌入的那篇](/zh/blog/zh/sentence-transformers-korean-rag-embedding-guide-2026)里重看分块大小，也正是为此。

## 这次测量没能触及的地方

老实写下局限。第一，token数只是成本的一个维度。同样的token在不同模型单价不同，输入和输出单价也不同。这篇只测了"有多少token"，"那到底多少钱"得代入你自己的模型和套餐。

第二，token多并不一定吃亏。CJK把同样的信息装进更少的字符里。韩语版字符比英语少却意思相同，就是证据。token效率和信息密度是两码事。

第三，我的语料是技术博客，正文里混了大量英语代码、专有名词和技术术语。纯粹的日常韩语散文，倍率可能更大。所以这个1.38倍是"技术文档基准、我的环境值"，不是普适常数。若想更深入分词器行为，我那篇[讲BPE量化与合并的文章](/zh/blog/zh/llama-cpp-iq-quantization-merge)是相邻的起点。

第四，模型一直在变。下一代分词器若再扩充CJK词汇，这个差距会再次收窄。所以这篇真正的结论不是"韩语是1.38倍"。而是"别估算，用你自己的模型分词器跑你自己的文本"。代码全在上面。接到你自己的语料上，十分钟就够。

为什么按token去复现和测量输出的这种感觉很重要，我在[用temperature和seed测输出可复现性的实验](/zh/blog/zh/llm-determinism-temperature-seed-experiment)里也是同一脉络处理的。用LLM这件事，归根到底是数token。在开始前就用数字知道这个计数单位因语言而异，和事后才发现，差别会写进你每月的账单。
