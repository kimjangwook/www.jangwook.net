오늘 샌드박스 실험을 준비하다가 예상하지 못한 것을 발견했다. Gemini API 모델 목록을 조회했더니 `gemini-3.5-flash`가 이미 배포되어 있었다. 공식 발표를 본 기억이 없어서 처음에는 잘못 읽은 줄 알았다.

확인해보니 실제로 호출 가능했다. 그래서 원래 계획하던 작업을 미루고, 현재 쓸 수 있는 Gemini 모델 4개를 동일 조건에서 전부 측정해봤다. 이 글은 그 실측 결과다.

## 측정 환경과 방법론

먼저 방법론을 솔직하게 밝힌다. 완벽한 벤치마크가 아니다.

```bash
# 실험 환경
Node.js v22.22.0
@google/generative-ai (최신)
측정일: 2026-05-24
프롬프트: "List 5 practical use cases for AI APIs in modern web applications. One sentence each."
입력 토큰: 19〜22 (모델마다 토크나이저 차이)
측정 횟수: 2회 평균
```

짧은 프롬프트, 2회 평균이니 통계적으로 신뢰하기에는 부족하다. 네트워크 상태, 서버 부하, 리전에 따라 결과는 달라진다. 하지만 "대략 이 모델이 이 정도 수준"이라는 감각을 잡는 데는 충분하다.

측정한 모델:
- `gemini-2.5-flash-lite` — Flash 라인의 보급형
- `gemini-2.5-flash` — 현재 대부분의 가이드에서 추천하는 기본 모델
- `gemini-2.5-pro` — 고성능 추론 모델
- `gemini-3.5-flash` — 오늘 발견한 최신 모델

## 속도 측정 결과: Flash-Lite가 예상보다 훨씬 빠르다

```
=== Gemini API Benchmark (2026-05-24 실측) ===

[gemini-2.5-flash-lite]
  Total: 2,447ms | TTFT: 1,981ms
  Input: 19 tok | Output: 159 tok
  Est. TPS: 65.0

[gemini-3.5-flash]
  Total: 5,783ms | TTFT: 5,103ms
  Input: 19 tok | Output: 186 tok
  Est. TPS: 32.2

[gemini-2.5-flash]
  Total: 6,334ms | TTFT: 5,849ms
  Input: 19 tok | Output: 170 tok
  Est. TPS: 26.8

[gemini-2.5-pro]
  Total: 11,931ms | TTFT: 11,140ms
  Input: 19 tok | Output: 159 tok
  Est. TPS: 13.3
```

Flash-Lite가 65 TPS로 압도적이다. Pro보다 4.9배 빠르고, 2.5 Flash보다 2.4배 빠르다. TTFT(첫 토큰까지 시간)도 Flash-Lite 1.9초 대 Pro 11.1초로, 체감 차이가 크다.

3.5 Flash는 흥미롭다. 2.5 Flash보다 약간 빠르고, 출력 토큰이 더 많았다(186 vs 170). 같은 프롬프트에서 더 풍부한 답변을 냈다는 뜻이다. 속도와 품질을 동시에 올린 셈인데, 공식 가격이 아직 확인되지 않아 비용 비교에는 한계가 있다.

Pro가 11초 걸리는 이유는 [thinking 모드가 기본 활성화되어 있기 때문이다](/ko/blog/ko/deep-thinking-ratio-llm-cost-optimization). 짧은 프롬프트에도 내부 추론 단계를 거치니 TTFT가 길어진다. 단순 작업에는 명백한 낭비지만, 복잡한 추론이 필요한 경우엔 이 지연이 의미가 있다.

## 비용 비교: 공식 가격 기준 (2026년 5월)

공식 문서에서 확인한 가격:

| 모델 | 입력 (1M 토큰) | 출력 (1M 토큰) | 컨텍스트 | 프리티어 |
|------|--------------|--------------|---------|---------|
| Gemini 2.5 Flash-Lite | **$0.10** | **$0.40** | 1M | ✓ |
| Gemini 2.5 Flash | $0.30 | $2.50 | 1M | ✓ |
| Gemini 3.5 Flash | ~$0.50* | ~$3.50* | 1M | 부분 |
| Gemini 2.5 Pro | $1.25 | $10.00 | 1M+ | 제한적 |

Flash-Lite는 Flash 대비 입력 비용이 67% 저렴하고, 출력은 84% 저렴하다. 속도도 2.4배 빠른데 가격까지 저렴하니, 단순 작업에 Flash를 쓰는 게 맞는가 의문이 생긴다.

Pro는 입력 기준으로 Flash-Lite보다 12.5배 비싸다. [기존 LLM 가격 비교 포스트](/ko/blog/ko/llm-api-pricing-comparison-2026-gpt5-claude-gemini-deepseek)에서 다룬 것처럼, 이 격차가 큰 것처럼 보여도 실제 사용 패턴과 출력 토큰 비율에 따라 총비용은 달라진다.

*Gemini 3.5 Flash 가격은 아직 공식 발표가 없다. API 응답에서 가격 정보를 확인하지 못했고, 여러 커뮤니티 추정치를 종합하면 2.5 Flash의 1.5〜2배 수준으로 보인다. 정확한 수치가 아님을 명시한다.

## 실제 사용 시나리오별 월 비용

이론적 가격보다 "내 서비스에 실제로 얼마가 나오나"가 중요하다. 세 가지 시나리오로 계산했다.

| 모델 | 월 예상 비용 |
|------|------------|
| Flash-Lite | **$130** |
| Flash | $650 |
| 3.5 Flash | ~$1,050 |
| Pro | $2,625 |

Flash-Lite와 Pro 사이 20배 차이다. 단순 FAQ 챗봇에 Pro를 쓰면 매달 $2,495을 낭비하는 셈이다.

**시나리오 2: 코드 리뷰 에이전트 (월 5만 요청, 입력 8,000 / 출력 2,000 토큰)**

| 모델 | 월 예상 비용 |
|------|------------|
| Flash-Lite | **$80** |
| Flash | $370 |
| 3.5 Flash | ~$600 |
| Pro | $1,500 |

긴 컨텍스트에서도 Flash-Lite가 가장 저렴하다. 하지만 코드 리뷰는 "정확도"가 중요한 작업이라, 단순 비용 비교로 모델을 고르면 나중에 후회할 수 있다.

**시나리오 3: RAG 문서 분석 (월 1만 요청, 입력 50,000 / 출력 1,000 토큰)**

| 모델 | 월 예상 비용 |
|------|------------|
| Flash-Lite | **$54** |
| Flash | $175 |
| 3.5 Flash | ~$280 |
| Pro | $725 |

RAG처럼 입력이 긴 경우, 캐싱을 적용하면 비용이 극적으로 줄어든다. 오늘 측정에서 840 토큰 입력 기준 Flash 요청 하나가 $0.001877이었다. 같은 시스템 프롬프트를 반복 사용하는 구조라면 Context Caching으로 입력 비용의 75% 이상을 줄일 수 있다.

## 어떤 모델을 선택해야 하는가

속도와 비용만 보면 Flash-Lite가 명백한 승자처럼 보인다. 하지만 실제로는 다르다.

**Flash-Lite를 써야 하는 경우:**
- 사용자 입력을 분류하거나 태깅하는 파이프라인
- 짧은 텍스트 생성 (제목, 요약, 키워드 추출)
- 응답 시간이 UX에 민감한 실시간 서비스
- 높은 QPS가 필요한 대량 처리 작업
- 비용 예산이 타이트한 초기 스타트업

솔직히 말하면, 많은 챗봇과 자동화 파이프라인은 Flash-Lite로 충분하다. Flash를 기본으로 쓰는 팀 중 상당수가 그 차이를 체감하지 못할 것이다.

**Flash를 써야 하는 경우:**
- 중간 복잡도의 지시 수행 (이메일 초안, 간단한 코드 생성)
- 대화 맥락을 유지하는 멀티턴 챗봇
- 이미지·동영상을 포함한 멀티모달 처리
- 품질과 비용 사이 기본 밸런스가 필요한 경우

Flash가 현실적인 기본값이다. Flash-Lite 대비 2.5배 느리고 3배 비싸지만, 더 복잡한 지시를 더 일관성 있게 수행한다. 내가 새 프로젝트를 시작한다면 Flash로 출발해서 비용 압박이 생길 때 Flash-Lite로 일부 파이프라인을 이전하는 방식을 택할 것 같다.

**3.5 Flash를 써야 하는 경우:**
가격이 확정되지 않아 아직 명확하게 추천하기 어렵다. 오늘 측정에서 2.5 Flash보다 빠르고 출력이 풍부했지만, 그게 모든 작업에서 동일하게 나타난다는 보장이 없다. 공식 가격 발표와 더 많은 실험 결과가 나오면 다시 판단할 예정이다.

**Pro를 써야 하는 경우:**
- 복잡한 코드베이스 분석 또는 아키텍처 리뷰
- 수학/과학 추론이 포함된 작업
- 긴 문서에서 미묘한 내용을 추출하는 RAG
- 정확도 손실이 비용보다 훨씬 비싼 B2B 사용 사례

Pro를 일반 챗봇에 쓰는 건 낭비다. 하지만 [AI 에이전트 비용 구조를 분석해보면](/ko/blog/ko/ai-agent-cost-reality), 잘못된 모델 선택보다 잘못된 에이전트 설계가 더 큰 비용 폭탄이 되는 경우가 많다. Pro를 써야 하는 곳에 Flash-Lite를 써서 정확도가 떨어지면, 재처리 비용이나 인간 검토 비용이 더 나올 수 있다.

## 비용 최적화 3가지 레버

모델 선택만이 비용을 줄이는 방법이 아니다.

**1. Context Caching**

동일한 시스템 프롬프트나 문서를 반복 사용하는 구조라면, Context Caching이 가장 강력한 레버다. Google 공식 문서 기준 캐시 히트 시 입력 비용의 75%를 절감한다. 오늘 측정에서 840 토큰짜리 긴 컨텍스트 요청을 1,000회 보낼 경우, 캐싱 없이 $1.877, 캐싱 적용 시 실제 절감분은 캐시 비율에 따라 다르지만 이론상 최대 75% 절감 가능하다.

```javascript
// Context Caching 적용 예시 (Gemini API)
const cache = await cacheManager.create({
  model: 'gemini-2.5-flash',
  contents: [{ role: 'user', parts: [{ text: systemDocument }] }],
  ttlSeconds: 3600,
});

const model = genAI.getGenerativeModelFromCachedContent(cache);
```

**2. Batch API**

비실시간 처리(일괄 분석, 야간 처리)는 Batch API를 쓰면 50% 할인된다. 월 $1,000짜리 작업이 $500으로 줄어든다. Flash-Lite + Batch API 조합이면 Pro 단독 대비 10배 이상 비용을 줄일 수 있는 경우도 있다.

**3. 티어 믹싱**

단일 모델로 모든 작업을 처리하지 않는 것이 핵심이다. 분류·라우팅·요약은 Flash-Lite, 핵심 생성 작업은 Flash, 복잡한 추론은 Pro로 나누는 구조가 가장 비용 효율이 높다. 이 패턴은 실제로 Claude API에서도 Haiku→Sonnet→Opus 계층화로 적용할 수 있다.

## 모델별 응답 스타일 관찰: 같은 프롬프트, 다른 결과

오늘 측정에서 속도와 비용 외에 응답 텍스트 자체도 살펴봤다. 같은 프롬프트("AI API의 실용적 사용 사례 5가지를 한 문장씩")를 줬을 때 모델마다 답변 방식이 달랐다.

**Flash-Lite**는 가장 간결했다. 번호를 붙이고 각 항목을 1문장으로 딱 끊었다. 지시를 글자 그대로 따른다는 느낌이다. 불필요한 서론도 없고, 결론 요약도 없다. 분류 파이프라인에서는 이런 간결함이 오히려 유리하다. JSON 파싱 대상 텍스트를 만들 때 부가 설명이 많으면 후처리가 복잡해진다.

**Flash**는 Flash-Lite보다 약간 풍부한 표현을 썼다. 항목마다 구체적인 예시가 붙거나, "예를 들어 ChatGPT 스타일의 서비스에서"처럼 맥락을 추가했다. 이 차이가 대부분의 사용자 대면 애플리케이션에서는 긍정적으로 작용한다.

**Pro**의 출력은 다른 모델과 다른 패턴을 보였다. 항목을 나열하기 전에 짧은 도입 문장을 넣고, 각 항목의 설명이 좀 더 분석적이었다. 같은 "한 문장씩"이라는 지시에도 불구하고 일부 항목에서 두 문장으로 넘어갔다. 이게 "더 좋다"가 아니라 "다르다"는 점을 강조하고 싶다. 지시를 엄격하게 따르는 것보다 더 나은 답변을 자율적으로 생성하려는 경향이 있는 것 같다. 특정 작업에서는 이게 강점이 되고, 다른 작업에서는 약점이 된다.

**3.5 Flash**는 오늘 측정에서 가장 많은 출력 토큰(186)을 냈다. 각 항목의 설명이 Flash보다 더 상세했고, 추가 맥락을 자연스럽게 포함했다. 프롬프트 지시를 따르면서도 독자에게 더 유용한 정보를 담으려는 균형이 느껴졌다. 물론 두 번의 측정으로 이 판단을 일반화하는 건 무리가 있다.

이 관찰에서 내가 얻은 교훈 하나: **모델을 선택할 때 "얼마나 빠른가"와 "얼마나 저렴한가" 외에 "지시를 얼마나 정밀하게 따르는가"도 고려해야 한다.** 분류·추출·구조화된 출력 생성에서는 지시 추종 정밀도가 핵심이다. 대화형 UX에서는 약간의 자율성이 오히려 사용자 만족도를 높일 수 있다.

## Flash에서 Flash-Lite로 마이그레이션할 때 체크리스트

모델을 바꾸는 건 설정 한 줄이지만, 그게 올바른 결정인지 검증하는 게 더 어렵다. Flash-Lite로 전환을 고려할 때 내가 확인하는 항목들이다.

**전환 전 반드시 확인:**

1. **현재 사용 중인 프롬프트 유형을 분류한다.** 모든 프롬프트가 동일한 복잡도가 아니다. 분류(A인가 B인가), 요약(3줄로 줄여라), 추출(이 텍스트에서 날짜를 꺼내라) 같은 작업은 Flash-Lite가 충분하다. 반면 "이 코드의 논리적 오류를 찾고, 수정 방안을 제시하고, 테스트 케이스도 작성하라"는 멀티스텝 작업은 Flash 이상이 필요할 수 있다.

2. **품질 기준을 숫자로 정한다.** "잘 되면 좋겠다"가 아니라, "정확도 95% 미만이면 리젝"처럼 측정 가능한 기준이 있어야 한다. 기준이 없으면 A/B 테스트 결과를 해석할 수 없다.

3. **A/B 테스트를 작은 트래픽으로 먼저 한다.** 전체 파이프라인을 바꾸기 전에 5〜10% 트래픽을 Flash-Lite로 라우팅해서 실제 품질과 비용을 측정한다. 이렇게 하면 사용자 영향 없이 데이터를 수집할 수 있다.

4. **오류율과 재시도 비용을 계산한다.** Flash-Lite가 특정 작업에서 더 많은 오류를 내거나 재생성이 필요하다면, 표면적인 단가 차이가 상쇄된다. 비용 계산은 항상 "재시도 포함 실제 비용"이어야 한다.

5. **사용자 피드백 채널을 열어둔다.** 자동화 파이프라인은 품질 저하를 알아채기 어렵다. 사용자가 직접 접하는 결과물이라면 피드백 메커니즘이 필수다.

```javascript
// 모델 라우팅 기본 패턴
function selectModel(taskType, complexityScore) {
  if (complexityScore < 0.3) {
    return 'gemini-2.5-flash-lite';  // 단순 분류, 추출
  } else if (complexityScore < 0.7) {
    return 'gemini-2.5-flash';       // 일반 생성, 요약
  } else {
    return 'gemini-2.5-pro';         // 복잡한 추론, 분석
  }
}
```

`complexityScore`를 어떻게 측정할지가 실제 구현의 핵심 문제다. 프롬프트 길이, 지시 수, 멀티스텝 여부를 조합해서 휴리스틱을 만드는 게 현실적인 출발점이다.

## Google 프리티어 전략: 개발 단계에서의 활용

Flash와 Flash-Lite 모두 프리티어가 있다. 정확한 한도는 달마다 변경되지만, 2026년 5월 기준으로 개인 개발자와 소규모 팀이 프로토타입을 만들기에는 충분한 수준이다.

프리티어로 할 수 있는 것:
- **프로토타입 개발**: 프로덕션 비용 걱정 없이 아이디어를 빠르게 검증할 수 있다
- **벤치마크 실험**: 오늘 내가 한 것처럼, 모델 간 비교를 비용 없이 할 수 있다
- **소규모 내부 도구**: 팀 내부에서 쓰는 자동화 도구 정도는 프리티어로 운영 가능한 경우가 많다

주의할 점은 **프리티어에서의 성능이 유료 티어와 다를 수 있다**는 것이다. Rate limit이 더 낮고, 서버 리소스 우선순위도 다를 가능성이 있다. 오늘 내 측정도 어느 티어에서 실행됐는지 정확히 알 수 없다.

프리티어 사용 중에 비용이 발생하지 않더라도, 프로덕션으로 전환할 때 같은 성능이 보장된다는 가정은 위험하다. 스케일 테스트는 유료 환경에서 따로 진행해야 한다.

## 다른 프로바이더와의 포지셔닝

Gemini 모델 내에서 비교를 했지만, 실제로는 OpenAI와 Claude의 해당 티어와 비교도 필요하다.

나의 관점으로 정리하면:

**비용 최우선이라면**: Flash-Lite는 Claude Haiku 4.5($1.00/1M input)보다 10배, GPT-5.4 Nano($0.20/1M input)보다 2배 저렴하다. 순수 비용 관점에서 Google이 가장 공격적인 하단 티어 가격을 제공하고 있다.

**추론 능력이 중요하다면**: 나는 Gemini 2.5 Pro보다 Claude Sonnet 4.6이나 GPT-5.4가 복잡한 지시 수행에서 더 일관성이 있다고 본다. Gemini Pro는 수학/과학/코딩 벤치마크에서 강하지만, 창의적 생성이나 미묘한 지시 해석에서는 아직 Claude나 OpenAI에 비해 아쉬운 부분이 있다. 이건 내 주관적 경험이고, 모든 작업에 동일하게 적용되지 않는다.

**생태계 통합이 필요하다면**: Google Workspace, BigQuery, Vertex AI와 연동하는 프로젝트는 Gemini 선택이 자연스럽다. 반대로 Anthropic Claude Code로 개발하거나 OpenAI 에이전트 SDK를 쓰는 팀은 그 생태계 안에 있는 모델이 더 유리할 수 있다.

한 프로바이더에 올인하는 것보다 **작업 유형별로 최적 모델을 선택**하는 방향이 장기적으로 더 낫다. 벤더 lock-in을 피하려면 추상화 레이어(LangChain, LiteLLM 등)를 통해 모델을 교체할 수 있는 구조를 처음부터 설계해두는 게 좋다.

## 오늘 실험에서 솔직하게 못한 것들

몇 가지 한계를 명시한다.

첫째, **품질 비교를 하지 않았다.** 속도와 비용만 측정했고, 실제 응답 품질은 체계적으로 평가하지 못했다. "출력 토큰이 많다"가 "더 좋은 답변이다"를 의미하지 않는다. Flash-Lite가 65 TPS로 빠르지만, 복잡한 작업에서 답변 품질이 Flash의 80% 수준이라면 상황에 따라 다른 판단이 필요하다. 품질 평가는 별도의 포스트에서 구체적인 작업 유형별로 다뤄볼 생각이다.

둘째, **Gemini 3.5 Flash의 공식 가격을 확인하지 못했다.** 오늘 API 목록에서 발견했지만, 공식 가격 페이지에서 별도 항목을 찾지 못했다. 일부 커뮤니티 데이터를 참고했지만 정확도를 보증할 수 없다. 혹시 이 모델을 이미 써보신 분이 계시다면 코멘트를 달아주시면 좋겠다.

셋째, **한 번의 측정은 참고용이다.** Google의 서버 상태, 내 네트워크 지연, 프롬프트 특성에 따라 결과가 달라진다. 같은 실험을 다른 날, 다른 시간대에 반복하면 다른 숫자가 나올 수 있다. 실제 프로덕션 결정을 내리기 전에 자신의 워크로드로 직접 측정하는 것을 권장한다.

넷째, **Thinking 비활성화 설정을 테스트하지 않았다.** Pro의 느린 TTFT는 Thinking 모드 때문인데, `thinkingConfig: { thinkingBudget: 0 }`으로 비활성화하면 속도가 크게 달라진다. 이 설정을 사용하면 Pro의 가격에서 추론 모델의 성능을 포기하는 셈이지만, 특정 상황에서는 실용적인 선택이 된다.

## 내 결론

오늘 실험을 정리하면 이렇다.

Flash-Lite는 "보급형 모델"이라는 이름이 무색하게 빠르고 저렴하다. 단순 분류, 라우팅, 짧은 생성 작업에 Flash를 기본으로 쓰는 팀이라면 일부 파이프라인을 Flash-Lite로 전환하는 것을 진지하게 검토할 만하다. 오늘 측정에서 65 TPS와 $0.10/1M input은 현재 주요 프로바이더 중 가장 공격적인 하단 가격이다.

3.5 Flash는 나의 관심을 끌었다. 2.5 Flash보다 빠르고 풍부한 출력을 냈는데, 공식 문서가 없다는 게 아직 베타 단계임을 시사한다. 조금 더 지켜볼 예정이다. 만약 가격이 Flash 수준으로 책정된다면 상당히 매력적인 선택지가 될 수 있다.

Pro는 단가가 높지만 그게 낭비인가 아닌가는 작업의 성격에 달려 있다. 복잡한 추론이 필요한 B2B 작업에서 Pro의 11초 TTFT와 높은 단가는 충분히 정당화된다. 하지만 단순 작업에 Pro를 쓰는 건, 드라이버를 박아 넣을 때 해머를 쓰는 것과 비슷하다.

Gemini API를 처음 시작하는 분께는 Flash로 출발해서 데이터를 보면서 조정하라고 권하겠다. Flash-Lite로 직접 뛰어드는 것보다 품질 기준을 먼저 세우고 나서 비용을 줄이는 순서가 낫다. 한 달만 운영하면서 실제 프롬프트 복잡도 분포를 파악한 뒤 모델 믹싱 전략을 수립하면, 대부분의 팀에서 30〜60% 비용 절감을 달성할 수 있다고 본다.

오늘 샌드박스 테스트에서 Gemini 3.5 Flash를 발견한 건 예상 밖의 수확이었다. 그 모델이 실제로 얼마나 쓸 만한지는 앞으로 더 많은 실험이 필요하지만, API 접근이 가능하다는 사실 자체가 의미 있다. Google이 모델 라인업을 얼마나 빠르게 확장하고 있는지, 그리고 개발자가 그 속도를 따라잡기 위해 얼마나 자주 벤치마크를 다시 해야 하는지를 잘 보여주는 사례였다.
